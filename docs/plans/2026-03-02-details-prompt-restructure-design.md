# 詳細作成プロンプト再構成 設計書

**日付**: 2026-03-02
**課題**: 詳細作成の出力品質が不安定（ページ間の重複、品質のバラつき）
**方針**: Anthropic公式ベストプラクティスに基づくプロンプト再構成（アプローチB）

---

## 1. 背景と診断

### 現在の問題
- ルール約20個、具体例ゼロ → AIが「何が正解か」を毎回推測している
- 同じ指示の重複（「結論→根拠」が3箇所に出現）
- XMLタグなし → 指示とデータの区別が曖昧
- データ配置が最適でない → 品質のバラつき

### Anthropicベストプラクティスとの乖離
| 推奨事項 | 現状 | 改善後 |
|----------|------|--------|
| XMLタグで指示・データを区分 | Markdownのみ | XMLタグ導入 |
| 具体例で品質基準を示す | 例ゼロ | 2件追加 |
| ルールは最小限に | 約20個 | 5個 |
| データ上部・指示下部 | 混在 | 分離 |

## 2. 新プロンプト構造

### システムプロンプト（details.ts）

```
<role>1文で役割を定義</role>

<rules>
  5個の核心ルールのみ
</rules>

<format>
  マスタースライド別の必須フィールド（簡潔版）
</format>

<examples>
  例1: 良い出力（3ページ分）— 品質の基準線
  例2: 失敗→正解の対比 — 具体性の基準
</examples>
```

### ユーザープロンプト（route.ts で構成）

```xml
<input>
  <output_type>スライド or ドキュメント</output_type>
  <structure>ページ構成JSON</structure>
  <confirmed_messages>確定済みメッセージ一覧</confirmed_messages>
  <research_memo>リサーチ内容</research_memo>
  <knowledge_base>RAG参考表現</knowledge_base>
</input>

全ページの詳細コンテンツをJSON形式で作成してください。
```

## 3. 核心ルール5個

| # | ルール名 | 内容 |
|---|---------|------|
| 1 | スコープ制約 | 各ページは確定済みメッセージの「展開」のみ。範囲外の内容を追加しない |
| 2 | 領域独立 | 同じデータ・主張を複数ページで繰り返さない。各ページは固有の役割に集中 |
| 3 | 具体性 | リサーチメモの数値・事例を使う。「効率化を実現」のような抽象表現は禁止 |
| 4 | 量の上限 | 箇条書き最大4点×40字、スライド全体200字以内 |
| 5 | マスター別必須 | COVER→title+subtitle、CONTENT_1COL→title+body/bullets など |

### 削除するもの（例で代替）
- 「結論→根拠の順で記述」
- 「箇条書きだけで完結させず body で補足」
- 「アイコンはIconify MDI形式」（formatセクションに1行のみ）
- 「参考表現の文体を参考に」
- 「JSONのみ出力」（タスク指示に移動）

## 4. 具体例の設計

### 例1: 良い出力（品質基準）

3ページ分の完全なJSON出力を示す。以下を自然に実演:
- 結論→根拠の書き順
- body + bullets の使い分け
- ページ間で領域が重ならない書き方
- リサーチデータの具体的な埋め込み方

### 例2: 失敗→正解の対比（具体性基準）

```
BAD:  bullets: [{ text: "効率化を実現" }]
GOOD: bullets: [{ text: "月次レポート作成を8時間→1.5時間に短縮" }]
```

## 5. 変更対象ファイル

| ファイル | 変更内容 |
|----------|----------|
| `lib/ai/prompts/details.ts` | DETAILS_PROMPT: XMLタグ構造に全面書き直し |
| `lib/ai/prompts/details.ts` | DOCUMENT_DETAILS_PROMPT: 同上（ドキュメント版） |
| `app/api/ai/details/route.ts` | ユーザープロンプト構成部分をXMLタグ + データ上部配置に変更 |

### 変更しないもの
- `revise/route.ts`, `revise-all/route.ts`, `apply-feedback/route.ts`（今回のスコープ外）
- UIコンポーネント（変更不要）
- JSON解析ロジック（変更不要）

## 6. 期待される効果

- **重複の減少**: ルール2（領域独立）+ 例1で「重ならない書き方」を実演
- **バラつきの減少**: 例が品質の「アンカー」となり、出力のばらつき幅を狭める
- **プロンプトトークンの削減**: ルール削減により推定10-15%減
