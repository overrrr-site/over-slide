# HIG（ヒューマンインターフェースガイドライン）デザイン原則ガイド

> Apple HIGの主要原則をWebアプリ設計に活かすためのコンテキスト文書
> 作成日：2026-03-01

---

## 1. HIGとは

HIG（Human Interface Guidelines）は、Appleが定める**デザイン原則と推奨事項の体系**。iOS、macOS、watchOS、visionOSなどApple全プラットフォームを横断して、使いやすく一貫性のあるUIを作るためのガイドライン。

本プロジェクトはWebアプリだが、HIGの原則はプラットフォームを問わず**良いUI設計の普遍的な指針**として活用できる。

---

## 2. 4つの核心原則

### 原則1：Clarity（明瞭さ）

**読みやすく、理解しやすいこと。**

- テキストはどのサイズでも読みやすくする
- アイコンは意味が明確で、一目で分かるものにする
- 装飾は機能を損なわない範囲にとどめる
- 情報の優先度を視覚的な階層で表現する（大きさ、太さ、色）
- 1画面に詰め込みすぎず、最も重要な要素にフォーカスする

**Webアプリへの適用：**
- 見出し・本文・補足の3段階で文字サイズを統一する
- ボタンやリンクのラベルは「何が起きるか」を明確に書く
- エラーメッセージは「何が問題か」「どうすれば直るか」を具体的に伝える

### 原則2：Deference（控えめさ）

**UIは主役ではなく、コンテンツが主役。**

- UIの装飾を最小限にして、ユーザーのデータや作業内容を際立たせる
- 背景やボーダーは控えめに、コンテンツの邪魔をしない
- アニメーションはスムーズで意味のあるものだけに限定する
- ツールバーやメニューは必要なときだけ目立たせる

**Webアプリへの適用：**
- サイドバー・ヘッダーは控えめな色使いで、作業エリアを最大化する
- モーダルやポップアップは本当に必要な場合のみ使う
- 企画書のプレビュー画面ではUI要素を最小化し、コンテンツに集中させる

### 原則3：Depth（奥行き）

**階層構造を視覚的に伝え、「今どこにいるか」を分かりやすくする。**

- 視覚的なレイヤーで前景・背景の関係を表現する
- 画面遷移はスムーズなアニメーションで「移動の感覚」を伝える
- 影やぼかしで要素間の重なりを表現する
- 戻る操作が直感的にできるようにする

**Webアプリへの適用：**
- パンくずリストやステップバーで現在位置を常に表示する
- モーダルは背景をぼかして「上に重なっている」感覚を出す
- 一覧 → 詳細の遷移でアニメーションを使い、階層の移動を体感させる

### 原則4：Consistency（一貫性）

**アプリ全体で統一された体験を提供する。**

- 同じ操作は同じ見た目・同じ場所で提供する
- 用語やラベルを統一する（例：「保存」と「セーブ」を混在させない）
- コンポーネントの見た目と振る舞いをアプリ全体で揃える
- プラットフォームの標準的なパターンに沿う

**Webアプリへの適用：**
- ボタンのスタイル（色、サイズ、角丸）を全画面で統一する
- 企画書側と見積側で共通のレイアウトパターンを使う
- 「作成」「編集」「削除」などの操作は同じアイコン・同じ位置に配置する

## 3. 基盤要素（Foundations）

### 色（Color）

- **目的を持った色使い**：色は見た目だけでなく、意味を伝えるために使う
- **インタラクティブ要素の区別**：クリックできるものと静的な情報を色で区別する
- **状態の表現**：成功（緑）、警告（黄）、エラー（赤）、情報（青）を一貫させる
- **同じ色に複数の意味を持たせない**：混乱を防ぐため、1つの色は1つの意味に限定する
- **アクセシビリティ**：色だけに頼らず、形やテキストでも情報を伝える（色覚多様性への配慮）
- **コントラスト比**：テキストと背景のコントラスト比を4.5:1以上確保する

### 文字（Typography）

- **読みやすさ最優先**：本文は14px以上、行間は1.5〜1.8
- **階層の表現**：見出し・小見出し・本文・注釈の4段階でサイズを統一する
- **フォントファミリーは2つまで**：多すぎると散漫になる
- **太さ（ウェイト）で強弱**：太字で重要な情報を際立たせる
- **行の長さ**：1行あたり40〜80文字が読みやすい

### レイアウト（Layout）

- **余白（ホワイトスペース）を恐れない**：詰め込まず、要素間に十分な余白を取る
- **整列（アライメント）**：左揃えを基本に、要素の開始位置を揃える
- **グリッドシステム**：8pxまたは4pxの倍数で余白とサイズを統一する
- **レスポンシブ**：画面サイズに応じてレイアウトが自然に調整される
- **視線の流れ**：左上から右下へ自然に視線が流れる配置にする

### アクセシビリティ（Accessibility）

- **操作手段の多様性**：マウス、キーボード、スクリーンリーダーで操作できるようにする
- **フォーカスの可視化**：キーボード操作時にフォーカス位置を明確に表示する
- **代替テキスト**：画像やアイコンには代替テキストを付ける
- **十分なタップ/クリック領域**：ボタンやリンクは最低44×44pxの操作領域を確保する
- **動きの制御**：アニメーションは reduced-motion 設定で無効化できるようにする

## 4. ナビゲーションパターン

### 階層型ナビゲーション

一覧から詳細へ、段階的に深く進むパターン。

```
プロジェクト一覧 → プロジェクト詳細 → ページ詳細
```

- **戻る操作**が明確であること
- **パンくずリスト**で現在位置を表示
- 本プロジェクトの基本パターンとして採用

### フラット型ナビゲーション

トップレベルのカテゴリ間を横に移動するパターン。

```
ダッシュボード ←→ 企画書 ←→ 見積 ←→ 管理
```

- ヘッダーのナビゲーションバーで実現
- どこからでもトップレベルに戻れる
- 本プロジェクトのグローバルナビゲーションとして採用

### コンテンツ駆動型ナビゲーション

コンテンツ自体がナビゲーションの起点になるパターン。

- 関連するオブジェクトへのリンクをコンテンツ内に配置
- 例：ブリーフシート内から該当するリサーチメモへ直接遷移

---

## 5. Webアプリへの応用まとめ

| HIG原則 | Webアプリでの具体的な実践 |
|---------|------------------------|
| Clarity | 文字の階層を統一、ボタンラベルを明確に、エラーは具体的に |
| Deference | 作業エリア最大化、装飾最小限、コンテンツにフォーカス |
| Depth | ステップバーで現在位置表示、モーダルの重なり表現、遷移アニメーション |
| Consistency | コンポーネント統一、用語統一、企画書側と見積側で共通パターン |
