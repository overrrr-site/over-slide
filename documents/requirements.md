## 1. プロジェクト概要

### 1.1 目的

OVER Inc. の営業・制作業務を支援する社内ツールを、1つのNext.jsアプリケーションとして統合する。

| ツール | 目的 | 主な利用者 |
|--------|------|-----------|
| **企画書作成システム** | AIを活用して社外クライアント向け企画提案書の作成を効率化・品質標準化する | チームメンバー（数名） |
| **ドキュメント作成機能** | 企画書作成と同一ワークフローで、情報を構造化したWordドキュメント（ディスカッションペーパー等）を生成する | チームメンバー（数名） |
| **見積作成ツール** | 案件タイプ別の項目サジェストと原価可視化により、精度の高い見積書を効率よく作成する | 営業担当 2名 |

### 1.2 統合の方針

- **モノリポ構成**：1つのNext.jsプロジェクトに全ツールを収める
- **認証・DBを共通化**：Supabase Auth・PostgreSQLを全ツールで共有
- **デプロイを一元化**：Vercel 1箇所でホスティング
- **ルーティングで分離**：`/proposals`（企画書・ドキュメント）、`/quotes`（見積）として独立して動作
- **将来の連携に備える**：DBスキーマ設計の段階から案件IDで紐付け可能な構造にしておく

### 1.3 解決する課題

**企画書作成システム（既存）**
- 企画書作成に時間がかかりすぎる
- 人によって構成・品質にばらつきがある
- リサーチ→構成→詳細→デザインの工程が属人化している
- レビューの観点が統一されていない

**見積作成ツール（新規追加）**
- 項目の抜け漏れが多く、後から追加費用が発生し利益を圧迫している
- 工数の見積もりが感覚的で根拠がなく、精度にばらつきがある
- 原価（内部工数・外注費）が可視化されておらず、粗利を意識した見積作成が難しい
- 必要な但し書き・免責事項の記載が不十分で、スコープトラブルのリスクがある
- 過去の見積知見が蓄積されず、毎回ゼロから作成している

| インフォグラフィック | SVG直接生成（Claude出力 → PptxGenJS埋め込み） | Phase 1はSVG。品質不足時にSatori追加 |

### 2.3 AIモデル使い分け（企画書作成システム）

| 工程 | モデル | 理由 |
|------|--------|------|
| 工程0：ディスカッション | **Opus 4.6** | 仮説の深掘り・壁打ちには高い対話力・推論力が必要 |
| 工程1：リサーチ | Sonnet 4.5 | 情報収集・要約はSonnetで十分。コスト効率重視 |
| 工程2：構成作成 | **Opus 4.6** | ストーリーラインの論理構成は品質が最重要 |
| 工程3：ページ別詳細 | Sonnet 4.5 | 構成が確定済みのため、それに沿った生成はSonnetで十分 |
| 工程4：デザイン化 / ドキュメント生成 | Sonnet 4.5 | PptxGenJS用JSON/SVGコード生成、またはdocx構造生成 |
| 工程5：レビュー | **Opus 4.6** | 論理の飛躍・情報の抜け漏れ検出は高い推論力が必要 |
| 工程6：修正 | Sonnet 4.5 | 指示に沿った修正適用 |
| ナレッジ解析 | Sonnet 4.5 | 過去企画書の解析・ベクトル化 |

## 3. 機能要件

### 3.1 共通機能

#### 認証・ユーザー管理

- Supabase Authによる招待制ログイン（メール＋パスワード）
- チームメンバーのみアクセス可能
- 全ツール（企画書・ドキュメント・見積）を同一ログインで利用可能
- ロール管理：将来拡張のため `role` カラムを保持（現時点は全員同一権限）

#### ナビゲーション

- ヘッダーに「企画書を作る（/proposals）」「見積を作る（/quotes）」のナビゲーション
- ダッシュボード（`/`）：全ツールのサマリーを表示

---

### 3.2 企画書・ドキュメント作成システム（/proposals）

#### ワークフロー（固定6+1工程・出力タイプで工程4が分岐）

全工程はプロジェクト単位で管理され、途中保存・再開が可能。プロジェクト作成時に出力タイプ（スライド / ドキュメント）を選択し、工程4のみ処理が分岐する。工程0〜3・5・6は完全共通。

```
[0. ディスカッション] → [1. リサーチ] → [2. 構成作成] → [3. ページ別詳細]
                                                                ↓
                                        ┌───────────────────────┤
                                        ↓                       ↓
                              [4a. スライドデザイン化]   [4b. Wordドキュメント生成]
                                 （PptxGenJS）              （docx npm）
                                        ↓                       ↓
                              [5. レビュー] ←─────────────────→ [5. レビュー]
                                        ↓
                              [6. 修正・最終出力]
                                  ↑          │
                                  └──────────┘
                                レビュー不合格時ループ
```

---

#### 工程0：ディスカッション（Opus 4.6）

**目的：** ユーザーの仮説・アイデアの断片を対話で引き出し、提案の方向性を言語化・構造化する。AIがユーザーの意見に迎合せず、バランスの取れたフィードバックで「角のある強い企画」を作ることを重視する。

**4モード切替型ディスカッション：**

ユーザーの状況に応じてAIの振る舞いを切り替える4つのモードを提供。デフォルトは「引き出し」モードから開始し、ユーザーが手動で切り替えるか、AIの提案に基づいてモード遷移する。モード間は自由に行き来可能。

| モード | キー | 用途 | AIの振る舞い |
|--------|------|------|-------------|
| **引き出し** | `draw_out` | 考えがまとまっていない段階 | 深掘り質問で核を掘り起こす。5項目（クライアント情報・背景・仮説・ゴール・制約）を内部追跡 |
| **壁打ち** | `challenge` | アイデアの断片がある段階 | 強み/弱みをフラットに評価。改善提案を付ける。「この部分は強い。一方で〇〇の観点が弱い」 |
| **膨らませ** | `expand` | 方向性が見えた段階 | 具体例・事例・データを追加。ユーザーの核を土台に広げる。AIが主導権を取りすぎない |
| **まとめ** | `structure` | 材料が十分な段階 | 全体を提案の流れに構成。トーンに応じた構成（論理型/感情型/ハイブリッド）。ブリーフシート生成 |

**モード切替の仕組み：**

- **AI提案：** AIが応答内に `[MODE_SUGGEST:モード名]` マーカーを出力 → チャット内に切替カードとして表示。ユーザーがワンクリックで受け入れ/却下
- **手動切替：** チャット上部のモードセレクターで4モード間を自由に切替
- **行き来自由：** まとめモード中に壁打ちに戻るなど、制限なく切替可能

**引き出しモードの内部追跡（カバレッジトラッカー）：**

引き出しモード時、AIは以下の5項目のカバー状況を追跡し、応答内に `[COVERAGE:項目1,項目2]` マーカーを出力する。UIではカバー済み/未カバーをドットで表示。

| # | 項目キー | 引き出したい情報 |
|---|---------|-------------|
| 1 | `client_info` | 提案先のクライアント（業種、規模、担当者の立場など） |
| 2 | `background` | クライアントが抱えている課題や提案に至った背景 |
| 3 | `hypothesis` | ユーザーが考えている解決の方向性やアイデアの断片 |
| 4 | `goal` | 提案が成功した場合の理想の状態 |
| 5 | `constraints` | 予算・期間・技術的制約、NG事項 |

**AIのフィードバック方針（全モード共通）：**

- バランス型：良い点は認める、弱い点は明確に指摘する（迎合しない）
- 弱い点の指摘には必ず改善案をセットで提示（「こうすれば、もっとクライアントに刺さる」）
- 一度に一つの質問・指摘に絞る（短い応答を心がける）
- ユーザーの核となるアイデアを尊重しながら、より強い企画にすることを目指す

**ブリーフシートのトーン選択（まとめモード時）：**

まとめモード時に以下の3つのトーンから構成スタイルを選択可能。

| トーン | 構成スタイル |
|--------|-------------|
| **論理型** (`logical`) | 根拠→課題→解決策→効果 の論理構成 |
| **感情型** (`emotional`) | ストーリー→共感→ビジョン→行動 の感情訴求構成 |
| **ハイブリッド** (`hybrid`) | 論理の骨格にストーリーを載せる複合構成（デフォルト） |

**出力：ブリーフシート（自動生成・編集可能）**

```
■ ブリーフシート
──────────────────────────
クライアント：[社名/業種/担当者の立場]
背景・課題：[現状の課題を2〜3文で]
提案の方向性：[仮説・アイデアの要約]
ゴール：[成功の定義]
制約条件：[予算/期間/技術/NG事項]
リサーチで確認すべきこと：[ディスカッションで出た不確定要素]
構成の骨格案：[選択トーンに応じた提案フローの方向性]
──────────────────────────
```

**UI：**
- チャット上部：モードセレクター（4モードボタン横並び、現在モードをハイライト）
- 引き出しモード時：カバレッジトラッカー（5項目のカバー状況をドットで表示）
- まとめモード時：トーンセレクター（論理型/感情型/ハイブリッドの3択）
- メインエリア：チャットUI（自由入力）。AIのモード提案はカード形式でチャット内に表示
- 右サイドパネル：ブリーフシートのリアルタイムプレビュー
- フッター：ブリーフシート手動更新ボタン、ディスカッション完了ボタン

#### 工程1：リサーチ（Sonnet 4.5）

**入力：** 工程0のブリーフシート、テーマ・キーワード、手持ちデータ（アップロード）

**処理：**
- Claudeが入力をもとにWeb検索クエリを生成
- Tavily API経由で情報収集（1プロジェクトあたり最大20回検索）
- アップロードデータの読み取り・要約
- 収集情報を構造化して「リサーチメモ」を生成

**出力：** リサーチメモ（Markdown形式）。ユーザーが追記・編集可能

---

#### 工程2：構成作成（Opus 4.6）

**入力：** 工程0のブリーフシート、工程1のリサーチメモ、スタイルガイド（モードA）、類似案件の過去企画書（モードB・RAG上位3件）

**処理：**
- ブリーフシートの「提案の方向性」「ゴール」「制約条件」を前提として読み込む
- 過去企画書の構成パターンをベクトル検索で取得（RAG）
- Claudeがストーリーライン案と全ページ構成を生成

**出力：** 構成案。ユーザーがページ順序・メッセージを編集可能

**ページメッセージの確定（工程3進行前に必須）：**
- 各ページに「このページで伝えること（1文・30字以内）」をAIが初期生成
- ユーザーが確認・修正し、全ページ分を確定してから工程3へ進む
- 工程3はこの確定メッセージを「展開するだけ」の役割とし、内容の逸脱・膨らみを防ぐ

---

#### 工程3：ページ別詳細作成（Sonnet 4.5）

**入力：** 確定した構成、確定済みページメッセージ（工程2）、リサーチメモ、スタイルガイド（モードA）、類似案件の過去企画書（モードB・RAG上位3件）、出力タイプ（スライド / ドキュメント）

**処理：**
- 出力タイプに応じてClaudeの生成指示を切り替える
  - **スライド**：見出し・本文・データ・アイコン指定・インフォグラフィック指示・チャートデータ・スピーカーノートを生成
  - **ドキュメント**：見出し・本文（段落形式）・表・箇条書き・引用・補足注釈を生成。視覚的装飾よりも情報の構造化・論理展開を優先
- 出力形式：共通の構造化JSON（`type: "slide" | "document"` フラグで工程4の処理を制御）

**冗長性抑制プロンプト設計：**
- 「工程2で確定したページメッセージを展開すること。メッセージの範囲を超えた内容を追加しない」を明示
- スライド：「1ページの箇条書きは最大4点。1点あたり40字以内」
- ドキュメント：「1節の本文は最大400字。結論→根拠→補足の順で書く」
- スタイルガイドの文体・フレーズをプロンプトに自動差し込み

**出力：** ページ別の詳細コンテンツ。ユーザーがページ単位で編集・再生成可能

---

#### 工程4a：スライドデザイン化（Sonnet 4.5）※スライド選択時

**入力：** ページ別詳細コンテンツ（構造化JSON）、デザインテンプレート、インフォグラフィック指示

**処理：**
- PptxGenJSでスライド生成（Vercel API Route上で実行）
- インフォグラフィック：Claude (Sonnet) がSVGコードを生成 → PptxGenJSに埋め込み
- 生成されたpptxをSupabase Storageに保存

**出力：** pptxファイル（ダウンロード可能）、HTMLプレビュー

---

#### 工程4b：Wordドキュメント生成（Sonnet 4.5）※ドキュメント選択時

**入力：** ページ別詳細コンテンツ（構造化JSON）

**処理：**
- `docx`（npm）でWord文書を生成（Vercel API Route上で実行）
- 構造化JSONの各ページを章・節として変換
- 生成されたdocxをSupabase Storageに保存

**ドキュメントスタイル仕様：**

| 要素 | 仕様 |
|------|------|
| ページサイズ | A4（11,906 × 16,838 DXA）|
| 余白 | 上下左右 25mm |
| 本文フォント | Arial 12pt |
| 見出し1 | Arial 16pt bold |
| 見出し2 | Arial 14pt bold |
| 見出し3 | Arial 12pt bold |
| 目次 | 自動生成（HeadingLevelベース） |
| ページ番号 | フッター中央に自動挿入 |
| 表 | グレー罫線、ヘッダー行にシェーディング |
| 箇条書き | numbering configによる正規実装 |

**出力：** docxファイル（ダウンロード可能）

#### 工程5：レビュー（Opus 4.6）

**入力：** ページ別詳細コンテンツ、ブリーフシート、ナレッジベース（モードB・RAG上位3件。レビュー判断基準テキストを含む）

**レビュー観点（優先度順）：**
1. 提案全体の構造と論理構成（結論→根拠の因果関係、前提の飛躍がないか）
2. ページ間のストーリーの流れ（飛躍・重複の有無）
3. 各ページのメインメッセージの明確さ
4. 文章表現の品質
5. 意思決定に必要な情報の抜け漏れ

**出力（構造化）：**
- A. 総評（通りにくい理由 3〜5点）
- B. ページメッセージ一覧とストーリー評価
- C. 構造・論理の指摘（ページ番号付き）
- D. 情報の抜け漏れ
- E. 文章・表現の改善提案（元→改善案の対比）
- F. 最小改修プラン（優先度順）
- G. 確認質問
- H. 冗長性チェック（各ページの主張が1つに絞られているか / 工程2確定メッセージとのずれ / 同内容の言い換え・繰り返し / 削除しても意味が通じる文・箇条書き）

**レビュアープロファイルの参照：**
ナレッジベース（モードB）に「レビュー判断基準」タグで登録されたテキスト（架空案件を一巡して言語化した判断軸・NG観点等）をRAGで参照し、レビュープロンプトに差し込む。「登録者ならこう指摘する」視点でのレビューを実現する。

**出力：** 最終版ファイル（pptx or docx）のダウンロード、「完了」or「再レビュー」

#### 過去企画書ナレッジベース

ナレッジベースは用途別に2モードで運用する。アップロード時にユーザーがモードを選択し、同一パイプライン・同一テーブルで管理する。

**モードA：スタイル学習**
- 目的：文体・構成パターンの抽出 → スタイルガイドへの反映
- 処理：Claude (Sonnet) が解析し、構成パターン・文体特徴・よく使う表現を抽出 → `style_guide` テーブルに反映
- 参照タイミング：スタイルガイド経由で工程2・3のプロンプトに自動差し込み

**モードB：内容参照**
- 目的：過去の類似案件の企画書内容をRAGで参照し、構成・コンテンツの精度を高める
- 処理：内容をそのままチャンク分割 → ベクトル化して `knowledge_chunks` に保存
- アップロード時にユーザーがメタデータをタグ付け（業種・案件タイプ・クライアント規模など）
- 参照タイミング：工程2（構成作成）・工程3（詳細生成）・工程5（レビュー）でタグをもとに類似案件を検索し上位3件を参照

**共通仕様：**
- 対象ファイル：pptx, PDF（6〜15本）
- 管理画面からアップロード・タグ付け・解析状況の確認が可能
- `knowledge_docs.purpose` カラム（`style` / `content`）でモードを管理

#### スタイルガイド

解析結果を4カテゴリに構造化して可視化し、ユーザーが確認・修正・追記できる機能。工程2・3のプロンプトに自動差し込みされ、生成品質の個人最適化を実現する。

**4カテゴリ：**

| カテゴリ | 解析・表示する内容 | 例 |
|---------|-----------------|---|
| 構成パターン | 論理展開の型、平均ページ数、セクション構成の傾向 | 「結論ファースト型が8/10本」「課題→原因→解決策の3段構成が多い」 |
| 文体・トーン | 語尾の傾向、1文の平均文字数、断定度 | 「体言止めが多い（語尾の60%）」「1文あたり平均30字」 |
| 情報密度 | 1ページあたりの箇条書き数・図表使用率・数値引用頻度 | 「1ページあたり箇条書き3点が典型」「数値・データを必ず入れる傾向」 |
| よく使うフレーズ | 頻出表現・業界用語・定型フレーズ | 「〜を実現する」「〜を通じて」 |

**チューニングの2レベル：**

レベル1（AIの解析結果を承認・修正）：AIが自動抽出した特徴に対して「合ってる / 違う / 修正」を選択。「違う」の場合は正しい内容をテキストで上書き。

レベル2（手動ルールの追加）：AIが自動抽出できない暗黙のルールを自由記述で追加。（例：「クライアントが大企業の場合は丁寧語寄りにする」「競合比較スライドは必ず入れる」）

**プロンプトへの自動差し込み：**
- スタイルガイドの内容は工程2（構成作成）・工程3（詳細生成）のプロンプトに毎回自動挿入
- ユーザーがスタイルガイドを更新したら次回の生成から即反映

### 4.1 データセキュリティ・AI学習防止

**Anthropic APIのデータポリシー（確認済み）：**
- 商用API経由のデータは、モデル学習に一切使用されない（商用利用規約で明示保証）
- APIログ保持期間は7日間
- ZDR（Zero Data Retention）契約は不要と判断。標準ポリシーで十分

**本システムでの対策：**

| 対策 | 詳細 |
|------|------|
| API経由のみ利用 | Claude.aiの消費者向けプランは使用しない |
| APIキー管理 | Vercelの環境変数（Secrets）で管理 |
| Supabase RLS | 行レベルセキュリティでチームメンバー以外のアクセスを防止 |
| ファイル暗号化 | Supabase Storageのサーバーサイド暗号化を有効化 |
| 通信暗号化 | 全通信をHTTPS（TLS）で暗号化 |

| style_guide | スタイルガイド | id, user_id, composition_patterns (JSONB), tone (JSONB), information_density (JSONB), phrases (JSONB), custom_rules (JSONB配列), updated_at |
| template_settings | テンプレート設定 | id, color_scheme (JSONB), font_settings (JSONB) |

#### 見積作成ツール

| テーブル | 概要 | 主なカラム |
|---------|------|-----------|
| quotes | 見積ヘッダー | id, user_id, quote_number, client_name, project_name, project_types (JSONB配列), status, issued_at, expires_at, subtotal, tax, total, notes (JSONB), created_at, updated_at |
| quote_items | 見積明細 | id, quote_id, sort_order, category, name, unit_price, quantity, unit, amount, internal_days, assignee_type, outsource_cost |

**quotes テーブルの補足：**
- `project_types`：選択した案件タイプを配列で保持（例：`["web", "content"]`）
- `status`：`draft`（下書き）/ `submitted`（提出済み）/ `won`（受注）/ `lost`（失注）
- `notes`：備考・免責事項テンプレートのON/OFF状態とテキストをJSONBで保持

**quote_items テーブルの補足：**
- `assignee_type`：`director`（人月300万）/ `sales`（人月200万）
- `internal_days`：内部工数（人日）
- `outsource_cost`：外注費見込み（円）
- `amount`：`unit_price × quantity`（クライアント向け金額）

### 5.2 計算仕様（見積ツール）

| 項目 | 計算式 |
|------|--------|
| 1人日あたりの人件費（ディレクター） | 3,000,000 ÷ 20 = 150,000円 |
| 1人日あたりの人件費（営業） | 2,000,000 ÷ 20 = 100,000円 |
| 内部人件費（行） | internal_days × 人日単価 |
| 原価合計（行） | 内部人件費 ＋ outsource_cost |
| 売上合計 | Σ amount |
| 原価合計 | Σ（内部人件費 ＋ outsource_cost） |
| 粗利 | 売上合計 ー 原価合計 |
| 粗利率 | 粗利 ÷ 売上合計 × 100（%、小数第1位まで） |
| 消費税 | 売上合計 × 0.1（円未満切り捨て） |
| 合計 | 売上合計 ＋ 消費税 |

### 5.3 RLS（Row Level Security）ポリシー

- 全テーブルでRLS有効化
- チームメンバーのみ自チームのデータにアクセス可能
- `quotes`・`quote_items` も同様に `user_id` または チームスコープでアクセス制御

## 8. 主要なリスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| Claude APIのレスポンス品質のばらつき | 生成物の品質が不安定 | プロンプトの厳密な設計 + few-shot例の整備。レビュー工程で品質担保 |
| PptxGenJSのデザイン品質の限界 | 手作業ほどの仕上がりにならない | テンプレート固定で割り切り。凝った部分はSVGインフォグラフィックで対応 |
| docxのレイアウト崩れ | WordとGoogle Docsで表示が異なる場合がある | DXA単位でテーブル幅を固定。WidthType.DXA必須。早期に両環境でテスト |
| 既存スライドのデザイン移植精度 | 完全再現できない要素がある | 主要なレイアウト・カラーを優先し、細部は段階的に改善 |
| 見積PDFの印刷品質 | @media printでのレイアウト崩れ | 早期に印刷テストを実施。問題がある場合はjsPDFに切り替え |
| 粗利率の誤計算 | 誤った原価認識 | 計算ロジックを単体テストで検証。表示値に計算根拠のツールチップを付ける |
| Web検索APIのコスト | リサーチ量に比例してコスト増 | 1プロジェクトあたり検索20回の上限設定 |
| Vercelのサーバーレス制約 | pptx/docx生成がタイムアウト | 20ページ程度なら5秒以内で完了（検証済み想定）。大規模時はPro（60秒）で対応 |
