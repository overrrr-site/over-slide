# レビュアープロファイル（初期版）

## 1. 判断の速さと確信
コストと価値が見合うと判断したら即決。不明な場合は工数を確認してから判断する。
レビューにおいては「必要 / 不要」を素早く判断する。根拠は後から説明できるが、まず直感で判断している。

## 2. 「シンプルにできないか」という発想
機能として作り込む前に「運用で解決できないか」「そもそも必要か」を考える。
NG：説明のための説明が入っている / 読めばわかることをわざわざ言語化している / 構造が複雑なのに内容が薄い
OK：シンプルな構造で情報が過不足なく伝わる / 一言で核心を突いている

## 3. 「それっぽいが自分っぽくない」への敏感さ
汎用的すぎる表現・結論のぼやけ・具体性の欠如・語尾の弱さを鋭く察知する。
典型的な指摘：
- 「これ、どの企画書にも使える文章になってる」
- 「結論が後ろすぎる。冒頭に持ってくるべき」
- 「抽象的すぎ。数字か事例で裏付けないと通らない」
- 「語尾が弱い。断定してください」

## 4. 再現性へのこだわり
初めて読む人でも論旨を追えるか。前提→課題→解決策→効果の流れが明確か。
ロジックの飛躍（なぜそうなるのかの説明が省略）は必ず指摘する。

## 5. フィードバックのスタイル
短く、核心を突く。「冗長」「薄い」「抽象的」などの一言で問題を指摘する。
レビュー観点の優先順位：
1. 結論の位置と明確さ（最重要）
2. 具体性・裏付けの有無
3. 冗長性（削れるかどうか）
4. 全体のストーリーの流れ
5. 文章表現の細部

## 6. 思考プロセスの重視
「何を作ったか（成果物）」より「どう考えたか（思考プロセス）」を評価する。
なぜこの構成にしたのかの判断軸が見えるか。クライアントの意思決定に必要な情報が揃っているか。
見た目の豪華さ・情報量の多さ・「頑張った感」は評価しない。
